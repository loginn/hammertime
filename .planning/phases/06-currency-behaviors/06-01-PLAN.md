  <name>Task 1: Create base Currency Resource and folder structure</name>
  <files>models/currencies/currency.gd</files>
  <action>
Create `models/currencies/` directory and the base Currency Resource class.

Currency extends Resource with:
- `var currency_name: String` -- display name (e.g., "Runic Hammer")
- `func can_apply(item: Item) -> bool` -- validates whether this currency can be used on the item. Base implementation returns false (subclasses override).
- `func apply(item: Item) -> bool` -- attempts to apply the currency to the item. Calls can_apply() first; if false, returns false (currency NOT consumed). If true, calls _do_apply(item) and returns true (currency consumed). This is the template method that enforces CRAFT-09 (consumed only on success).
- `func _do_apply(item: Item) -> void` -- virtual method for subclasses to implement actual behavior. Base does nothing.
- `func get_error_message(item: Item) -> String` -- returns human-readable reason why can_apply() would fail. Base returns empty string.

This pattern ensures:
- CRAFT-07: Validation happens in can_apply() before any mutation
- CRAFT-08: Error messages via get_error_message()
- CRAFT-09: apply() only mutates and returns true when can_apply() passes

All functions must have explicit return type hints. Use Resource (not Node) per project convention.

  <action>
Create two Currency subclasses:

**RunicHammer** (models/currencies/runic_hammer.gd):
- `class_name RunicHammer extends Currency`
- `_init()`: set currency_name = "Runic Hammer"
- `can_apply(item: Item) -> bool`: returns true only if item.rarity == Item.Rarity.NORMAL
- `get_error_message(item: Item) -> String`: if item.rarity != NORMAL, return "Runic Hammer can only be used on Normal items" (satisfies CRAFT-08)
- `_do_apply(item: Item) -> void`:
  1. Set item.rarity = Item.Rarity.MAGIC
  2. Add 1-2 random mods total. Use `randi_range(1, 2)` for count. For each mod, randomly choose prefix or suffix (50/50 via `randi_range(0, 1)`), call item.add_prefix() or item.add_suffix(). If the chosen type fails (at limit or no valid affixes), try the other type. Magic limit is 1+1, so with 1-2 mods this always works.
  3. Call item.update_value() after all mods added

**ForgeHammer** (models/currencies/forge_hammer.gd):
- `class_name ForgeHammer extends Currency`
- `_init()`: set currency_name = "Forge Hammer"
- `can_apply(item: Item) -> bool`: returns true only if item.rarity == Item.Rarity.NORMAL
- `get_error_message(item: Item) -> String`: if item.rarity != NORMAL, return "Forge Hammer can only be used on Normal items"
- `_do_apply(item: Item) -> void`:
  1. Set item.rarity = Item.Rarity.RARE
  2. Add 4-6 random mods total. Use `randi_range(4, 6)` for count. For each mod, randomly choose prefix or suffix (50/50), call item.add_prefix() or item.add_suffix(). If chosen type fails, try the other type. If both fail (pool exhausted), stop adding mods. Rare limit is 3+3, so 4-6 mods will fit.
  3. Call item.update_value() after all mods added

Important: When adding mods, set rarity BEFORE calling add_prefix()/add_suffix() because those methods enforce rarity limits. If rarity is still NORMAL when you call add_prefix(), it will fail (NORMAL allows 0 prefixes).

All functions must have explicit return type hints.
