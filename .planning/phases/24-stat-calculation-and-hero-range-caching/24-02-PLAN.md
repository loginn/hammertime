2. Add a new method `calculate_damage_ranges()` after `calculate_dps()`:
```gdscript
func calculate_damage_ranges() -> void:
	"""Populate per-element damage ranges from equipped weapon and ring affixes."""
	# Reset to zero
	for element in damage_ranges:
		damage_ranges[element]["min"] = 0.0
		damage_ranges[element]["max"] = 0.0

	# Weapon contribution: base damage + weapon affixes
	if "weapon" in equipped_items and equipped_items["weapon"] != null:
		var weapon = equipped_items["weapon"]
		if weapon is Weapon:
			var all_affixes := weapon.prefixes.duplicate()
			all_affixes.append_array(weapon.suffixes)
			if weapon.implicit:
				all_affixes.append(weapon.implicit)
			var weapon_ranges := StatCalculator.calculate_damage_range(
				weapon.base_damage_min, weapon.base_damage_max, all_affixes
			)
			for element in weapon_ranges:
				damage_ranges[element]["min"] += weapon_ranges[element]["min"]
				damage_ranges[element]["max"] += weapon_ranges[element]["max"]

	# Ring contribution: ring has base_damage but no base_damage_min/max yet
	# Ring uses flat base_damage (no range fields), so pass it as both min and max
	if "ring" in equipped_items and equipped_items["ring"] != null:
		var ring = equipped_items["ring"]
		if ring is Ring:
			var all_affixes := ring.prefixes.duplicate()
			all_affixes.append_array(ring.suffixes)
			if ring.implicit:
				all_affixes.append(ring.implicit)
			var ring_ranges := StatCalculator.calculate_damage_range(
				ring.base_damage, ring.base_damage, all_affixes
			)
			for element in ring_ranges:
				damage_ranges[element]["min"] += ring_ranges[element]["min"]
				damage_ranges[element]["max"] += ring_ranges[element]["max"]
```

Note: Ring does not have base_damage_min/max (it uses a single base_damage int). Passing base_damage as both min and max is correct -- ring damage is deterministic at the base level. Ring affixes with add_min/add_max will still contribute range variance through the affix calculation.

3. Modify `calculate_dps()` to use the range-based formula instead of summing weapon.dps + ring.dps:
```gdscript
func calculate_dps() -> float:
	"""Calculate total DPS from per-element damage range averages."""
	# Sum average damage across all elements
	var total_avg_damage := 0.0
	for element in damage_ranges:
		var el_min: float = damage_ranges[element]["min"]
		var el_max: float = damage_ranges[element]["max"]
		total_avg_damage += (el_min + el_max) / 2.0

	# Apply speed multiplier from weapon
	var speed := 1.0
	if "weapon" in equipped_items and equipped_items["weapon"] != null:
		var weapon = equipped_items["weapon"]
		if weapon is Weapon:
			speed = float(weapon.base_speed)
			# Apply speed modifiers from affixes
			var all_affixes := weapon.prefixes.duplicate()
			all_affixes.append_array(weapon.suffixes)
			if weapon.implicit:
				all_affixes.append(weapon.implicit)
			var additive_speed_mult := 0.0
			for affix: Affix in all_affixes:
				if Tag.StatType.INCREASED_SPEED in affix.stat_types:
					additive_speed_mult += affix.value / 100.0
			speed *= (1.0 + additive_speed_mult)

	# Apply crit multiplier
	var crit_multiplier := StatCalculator._calculate_crit_multiplier(
		total_crit_chance, total_crit_damage
	)

	total_dps = total_avg_damage * speed * crit_multiplier
	return total_dps
```

Wait -- there's a design issue. The `total_crit_chance` and `total_crit_damage` are populated by `calculate_crit_stats()` which runs after `calculate_dps()` in `update_stats()`. But actually looking at `update_stats()`:
```
func update_stats() -> void:
    calculate_dps()
    calculate_defense()
    current_energy_shield = float(total_energy_shield)
    calculate_crit_stats()
```

The crit stats are calculated AFTER dps. So the `total_crit_chance` used in `calculate_dps()` would be stale (from the previous update_stats call). This is a pre-existing issue that already exists with the current code -- `calculate_dps()` uses `weapon.dps` which was calculated by Weapon.update_value() at crafting time, not from hero crit totals.

For the hero-level DPS calculation, we need the crit stats to be calculated first. Fix this by reordering `update_stats()`:

```gdscript
func update_stats() -> void:
	"""Recalculate all hero stats based on equipped items"""
	calculate_crit_stats()
	calculate_damage_ranges()
	calculate_dps()
	calculate_defense()
	current_energy_shield = float(total_energy_shield)
```

This ensures crit stats are available when calculate_dps() runs, and damage ranges are available when calculate_dps() runs.

Note: The old `calculate_dps()` summed `weapon.dps + ring.dps` where each `.dps` was pre-calculated by the item's `update_value()`. The new formula calculates DPS directly from ranges + speed + crit at the hero level. This is more correct because it uses the HERO's total crit stats rather than the per-item base crit stats. This may produce slightly different DPS numbers than before, which is expected and correct.

Do NOT modify weapon.gd or ring.gd -- the per-item `update_value()` and `dps` field continue to exist for item-level display (Phase 26 may use them).
Do NOT modify stat_calculator.gd (that was Plan 01).
