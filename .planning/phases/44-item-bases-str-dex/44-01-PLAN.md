# Plan 01: Create 21 Item Base Classes & Update Defense Calculations

## Overview

Create all 21 data-driven item base classes (6 weapons, 3 armors, 3 helmets, 3 boots, 3 rings) with tier-parameterized constructors, tier name tables, tier stat tables, archetype-appropriate valid_tags, and unique implicits. Also update the 3 defense base classes (armor.gd, helmet.gd, boots.gd) so total_defense reflects the primary defense stat rather than hardcoding armor.

## Reference Data

All tier names, stat tables, valid_tags, and implicit designs come from `44-RESEARCH.md`. Key patterns:

- Each class extends its slot base (Weapon, Armor, Helmet, Boots, Ring)
- Constructor takes `p_tier: int = 8` parameter
- TIER_NAMES dict maps tier -> display name (168 unique names total)
- TIER_STATS dict maps tier -> stat values
- Implicit values scale with tier via pre-computed min/max in TIER_STATS
- Defense items have NO implicit; defense comes from base_armor/base_evasion/base_energy_shield

<task id="44-01-03">
<title>Update defense base classes to support multi-archetype total_defense</title>
<details>
Update `armor.gd`, `helmet.gd`, and `boots.gd` to set total_defense as the sum of all computed defense stats rather than hardcoding `self.computed_armor`. Since each archetype base only sets ONE non-zero base defense stat, this effectively makes total_defense reflect the primary stat.

In each file, change the line:
```gdscript
self.total_defense = self.computed_armor
```
to:
```gdscript
self.total_defense = self.computed_armor + self.computed_evasion + self.computed_energy_shield
```

This works because:
- STR bases have base_armor > 0, base_evasion = 0, base_energy_shield = 0 -> total_defense = computed_armor
- DEX bases have base_evasion > 0, others = 0 -> total_defense = computed_evasion
- INT bases have base_energy_shield > 0, others = 0 -> total_defense = computed_energy_shield

Check `helmet.gd` and `boots.gd` for the same pattern and apply the same fix. They should have similar update_value() methods with `self.total_defense = self.computed_armor`.
</details>

Defense items have NO implicit (set implicit = null). Defense scaling comes from base_armor, base_evasion, or base_energy_shield fields.

**Pattern for defense items:**
```gdscript
class_name IronPlate extends Armor

const TIER_NAMES: Dictionary = {
    8: "Rusty Plate", 7: "Iron Plate", 6: "Steel Plate",
    5: "Tempered Plate", 4: "War Plate", 3: "Champion Plate",
    2: "Valiant Plate", 1: "Sovereign Plate"
}

const TIER_STATS: Dictionary = {
    8: {"armor": 5}, 7: {"armor": 10}, 6: {"armor": 18},
    5: {"armor": 28}, 4: {"armor": 40}, 3: {"armor": 55},
    2: {"armor": 72}, 1: {"armor": 92},
}

func get_item_type_string() -> String:
    return "IronPlate"

func _init(p_tier: int = 8) -> void:
    self.rarity = Rarity.NORMAL
    self.tier = p_tier
    self.item_name = TIER_NAMES[p_tier]
    self.valid_tags = [Tag.STR, Tag.DEFENSE, Tag.ARMOR]
    var s = TIER_STATS[p_tier]
    self.base_armor = s["armor"]
    self.base_evasion = 0
    self.base_energy_shield = 0
    self.base_health = 0
    self.computed_armor = 0
    self.computed_evasion = 0
    self.computed_energy_shield = 0
    self.computed_health = 0
    self.implicit = null
    self.update_value()
```
