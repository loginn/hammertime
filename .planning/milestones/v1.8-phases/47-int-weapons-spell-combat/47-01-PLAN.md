# Plan 01: INT Weapon Bases & Spell Stat Types

## Objective
Add 3 INT weapon bases (Wand, Lightning Rod, Sceptre) with 8 tier variants each, register them in the item serialization system, add them to the drop pool, and extend the spell damage pipeline to support per-element spell damage (spell_fire, spell_lightning) so that Lightning Rod and Sceptre implicits work correctly.

## must_haves
- [ ] Wand, LightningRod, and Sceptre classes exist with 8 tiers each, following the established weapon pattern
- [ ] Each INT weapon has non-zero base_spell_damage_min/max and base_cast_speed, plus small non-zero base_damage_min/max and base_attack_speed
- [ ] FLAT_SPELL_LIGHTNING_DAMAGE and FLAT_SPELL_FIRE_DAMAGE stat types exist in tag.gd
- [ ] StatCalculator.calculate_spell_damage_range() routes new stat types to spell_fire and spell_lightning elements
- [ ] Hero.spell_damage_ranges includes spell_fire and spell_lightning keys
- [ ] Hero.calculate_spell_damage_ranges() and calculate_spell_dps() handle multi-element spell damage
- [ ] Item registry (ITEM_TYPE_STRINGS and create_from_dict) includes all 3 INT weapons
- [ ] INT weapons appear in the gameplay_view drop pool

### Task 2: Extend StatCalculator for per-element spell damage

**Do:**
- In `models/stats/stat_calculator.gd`, modify `calculate_spell_damage_range()` to return a dictionary with three elements: `"spell"`, `"spell_fire"`, `"spell_lightning"` (all initialized to `{"min": 0.0, "max": 0.0}` except "spell" which gets the base_spell_min/max).
- Add routing logic: affixes with `FLAT_SPELL_FIRE_DAMAGE` in stat_types add to `"spell_fire"` element, affixes with `FLAT_SPELL_LIGHTNING_DAMAGE` add to `"spell_lightning"` element. Existing `FLAT_SPELL_DAMAGE` continues routing to `"spell"`.
- `INCREASED_SPELL_DAMAGE` percentage scales ALL three spell elements uniformly (apply the same `spell_pct` multiplier to all three elements' min/max).
- Add a helper `_get_spell_damage_element(tags: Array) -> String` that checks for Tag.FIRE -> "spell_fire", Tag.LIGHTNING -> "spell_lightning", default -> "spell". But note: for spell flat damage, we use stat_types (FLAT_SPELL_FIRE_DAMAGE, FLAT_SPELL_LIGHTNING_DAMAGE) not tags for routing, since there could be attack flat damage affixes that also have FIRE tag. So routing should be done by stat_type check, not tag check:
  - `FLAT_SPELL_FIRE_DAMAGE` in affix.stat_types -> add to "spell_fire"
  - `FLAT_SPELL_LIGHTNING_DAMAGE` in affix.stat_types -> add to "spell_lightning"
  - `FLAT_SPELL_DAMAGE` in affix.stat_types -> add to "spell"

**Files:** `models/stats/stat_calculator.gd`

<verify>
- calculate_spell_damage_range() returns a dictionary with "spell", "spell_fire", "spell_lightning" keys.
- Passing an affix with FLAT_SPELL_FIRE_DAMAGE stat type populates the spell_fire element.
- INCREASED_SPELL_DAMAGE scales all three elements.
</verify>
</task>
