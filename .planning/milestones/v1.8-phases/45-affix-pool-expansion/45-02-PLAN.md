# Plan 45-02: Fix Flat Damage Range Rolling + Display

## Objective

Broaden the flat damage range check in affix.gd from FLAT_DAMAGE-only to any affix with damage range params, and update forge_view.gd display to handle new flat stat types (spell damage, bleed, poison, burn). Without this fix, new flat affixes will silently fail to roll add_min/add_max values and display incorrectly in the UI.

## Context

- `affix.gd` lines 73 and 88 hardcode `Tag.StatType.FLAT_DAMAGE in self.stat_types` as the gate for damage range rolling. New flat affixes use FLAT_SPELL_DAMAGE, BLEED_DAMAGE, POISON_DAMAGE, BURN_DAMAGE -- they pass damage range params but currently won't roll add_min/add_max.
- The cleanest fix is removing the stat type gate entirely: if damage range params are provided (dmg_min_hi > 0 or dmg_max_hi > 0), roll the ranges regardless of stat type.
- `forge_view.gd` line 933 also hardcodes FLAT_DAMAGE for the "Adds X to Y Damage" display format. Must be broadened to recognize all flat damage stat types.
- `_get_affix_element_name()` currently maps FIRE/COLD/LIGHTNING tags to element names, defaulting to "Physical". It needs new mappings for SPELL, DOT+PHYSICAL (Bleed), DOT+CHAOS (Poison), DOT+FIRE (Burn).

### Task 2: Update Forge View Display for New Flat Damage Types

**File:** `scenes/forge_view.gd`
**Action:** Modify

**Change 1 (line 933, in `_format_affix_line()`):** Replace the FLAT_DAMAGE-only check:
```gdscript
	if Tag.StatType.FLAT_DAMAGE in affix.stat_types and (affix.add_min > 0 or affix.add_max > 0):
```
With a check that covers all flat damage stat types:
```gdscript
	var flat_damage_stats := [
		Tag.StatType.FLAT_DAMAGE,
		Tag.StatType.FLAT_SPELL_DAMAGE,
		Tag.StatType.BLEED_DAMAGE,
		Tag.StatType.POISON_DAMAGE,
		Tag.StatType.BURN_DAMAGE,
	]
	var has_flat_damage := false
	for stat in affix.stat_types:
		if stat in flat_damage_stats:
			has_flat_damage = true
			break
	if has_flat_damage and (affix.add_min > 0 or affix.add_max > 0):
```

**Change 2 (`_get_affix_element_name()` function):** Expand the element name mapping to handle new affix types. Replace the entire function body:
```gdscript
func _get_affix_element_name(tags: Array) -> String:
	if Tag.SPELL in tags:
		return "Spell"
	if Tag.DOT in tags:
		if Tag.CHAOS in tags:
			return "Poison"
		if Tag.FIRE in tags:
			return "Burn"
		if Tag.PHYSICAL in tags:
			return "Bleed"
		return "DoT"
	if Tag.FIRE in tags:
		return "Fire"
	if Tag.COLD in tags:
		return "Cold"
	if Tag.LIGHTNING in tags:
		return "Lightning"
	return "Physical"
```

The DOT check must come before the FIRE check, because Burn affixes have both DOT and FIRE tags -- we want "Burn" not "Fire" for DoT affixes.

<verify>
- forge_view.gd compiles without errors
- Flat Spell Damage affix displays as "Adds X to Y Spell Damage (TN)"
- Flat Bleed Damage prefix displays as "Adds X to Y Bleed Damage (TN)"
- Flat Poison Damage prefix displays as "Adds X to Y Poison Damage (TN)"
- Flat Burn Damage prefix displays as "Adds X to Y Burn Damage (TN)"
- Existing Physical/Fire/Cold/Lightning flat damage affixes still display correctly (regression check)
- Non-flat affixes still display as "Name: value (TN)" format
</verify>
</task>
