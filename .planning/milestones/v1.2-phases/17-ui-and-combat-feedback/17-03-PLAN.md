<task type="auto">
  <name>Task 1: Fix mouse_filter and Background sizing in gameplay_view.tscn</name>
  <files>scenes/gameplay_view.tscn</files>
  <action>
Fix two issues in the .tscn scene file:

**Issue 1: UIRoot and containers blocking mouse input**

Add `mouse_filter = 2` (IGNORE) to these nodes so clicks pass through to layer 0 navigation buttons:
- `UIRoot` (the full-screen Control under CombatUI CanvasLayer)
- `HeroHealthContainer`
- `PackHealthContainer`
- `PackProgressContainer`

Note: `FloatingTextContainer` already has `mouse_filter = 2` -- leave it as-is. Do NOT set mouse_filter on the ProgressBar nodes or Labels inside containers -- those are positioned with explicit offsets and don't cover the full screen, plus ProgressBars are non-interactive display widgets that don't consume input anyway.

In Godot .tscn format, add this property line to each node listed above:
```
mouse_filter = 2
```

**Issue 2: Background ColorRect invisible due to anchors under Node2D**

The Background node currently uses `anchors_preset = 15` with `anchor_right = 1.0` and `anchor_bottom = 1.0`. Since its parent GameplayView is a Node2D (which has no size), the anchor multipliers resolve against 0, making the ColorRect invisible.

Fix by replacing the anchor-based sizing with explicit pixel offsets matching the viewport (1200x700, but the bottom 100px is NavigationPanel so use 600):

Replace the Background node properties:
```
[node name="Background" type="ColorRect" parent="."]
offset_right = 1200.0
offset_bottom = 600.0
color = Color(0.1, 0.3, 0.1, 1)
mouse_filter = 2
```

Remove `anchors_preset = 15`, `anchor_right = 1.0`, and `anchor_bottom = 1.0` from Background. Add `mouse_filter = 2` to Background as well since it should not intercept clicks.
  </action>

<task type="auto">
  <name>Task 2: Handle CanvasLayer visibility explicitly in main_view.gd</name>
  <files>scenes/main_view.gd</files>
  <action>
CanvasLayer nodes do not inherit their parent's `visible` property. When `show_view()` sets `gameplay_view.visible = false`, the CombatUI CanvasLayer remains visible and rendering (and would still block input if mouse_filter weren't fixed in Task 1).

Fix by explicitly toggling the CombatUI CanvasLayer visibility in `show_view()`.

1. Add an @onready reference at the top of main_view.gd (after the existing @onready vars):
```gdscript
@onready var combat_ui: CanvasLayer = $GameplayView/CombatUI
```

2. In `show_view()`, after the existing match block (which sets view visibility and button disabled states), add a line to sync CombatUI visibility:
```gdscript
# CanvasLayer doesn't inherit parent visibility — sync explicitly
combat_ui.visible = (view_name == "gameplay")
```

Place this line after the match block but before `current_view = view_name`.

This ensures:
- When switching TO gameplay view: CombatUI becomes visible (bars render)
- When switching AWAY from gameplay view: CombatUI hides (no phantom input blocking, no rendering waste)
  </action>

2. `show_view()` contains `combat_ui.visible = (view_name == "gameplay")` after the match block
3. No other changes to the file's logic
  </verify>
  <done>CombatUI CanvasLayer visibility is explicitly managed: visible only when gameplay view is the active tab, hidden otherwise.</done>
